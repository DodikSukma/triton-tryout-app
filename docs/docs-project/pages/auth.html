Autentikasi

Mekanisme session-based authentication dan gateway middleware.

🍪
Mengapa Session-Based (bukan JWT)?

✅ Keuntungan Session

  • 🔒 Session bisa di-invalidate server-side (logout instan)
  • 🔒 Tidak ada token di client — lebih aman dari XSS
  • 🍪 Cookie HttpOnly — tidak bisa diakses JavaScript
  • 🔄 Auto-renew saat ada aktivitas (rolling session)
  • 🛑 Bisa paksa logout semua device dari server

⚠️ Trade-off

  • 🔌 Perlu database/Redis untuk menyimpan session
  • 🔄 Setiap request ke gateway butuh validasi ke auth-service
  • 🌐 Tidak stateless (berbeda dari REST pure)
  • 📦 Session store harus di-share antar instance
📋
Struktur Session

Data yang disimpan di session setelah login berhasil:

// Session data (auth-service, express-session)
{
  userId: "uuid",           // ID pengguna dari db_auth.users
  role:   "admin|guru|siswa",
  email:  "user@triton.id"
}

// Cookie yang dikirim ke browser
Set-Cookie: triton.sid=s%3A...; Path=/; HttpOnly; SameSite=Lax
⚠️ Cookie triton.sid wajib dikirim di setiap request ke API. Frontend menggunakan axios dengan withCredentials: true untuk memastikan ini.

🔀
Gateway Auth Middleware Flow

Setiap request ke API Gateway melalui proses berikut:

1

Request masuk ke API Gateway (port 4000)

Browser mengirim request dengan cookie triton.sid secara otomatis.

2

requireAuth middleware

Gateway fetch GET /auth/me ke auth-service dengan meneruskan cookie. Jika 401, return 401 ke client.

// auth.middleware.ts
const r = await fetch(`${AUTH_SERVICE_URL}/auth/me`, {
  headers: { cookie: req.headers.cookie ?? '' }
})
if (!r.ok) return res.status(401).json({ error: 'Unauthorized' })
3

Inject headers ke downstream request

Gateway menambahkan x-user-id dan x-user-role ke request sebelum di-proxy ke service target.

req.headers['x-user-id']   = data.userId
req.headers['x-user-role'] = data.role
4

requireRole middleware (opsional)

Beberapa route punya role check tambahan. Jika role tidak sesuai, return 403.

// Contoh: hanya admin yang bisa akses /users
app.use('/users', requireAuth, requireRole('admin'), proxy(...))
5

Request di-proxy ke service target

Service membaca x-user-id dan x-user-role dari header untuk otorisasi internal.
